# 系统缓存清理工具软件 V1.0 模块验收文档

## 1. 验收目的

本文档把需求拆分为八个可独立交接的模块，并为每个模块规定可重复执行的验收方法。开发 Agent 负责提交实现和自测证据，验收 Agent 根据本文档复核，不以代码作者的口头说明代替测试结果。

模块状态只有“未开始、开发中、待验收、通过、未通过、受阻”六种。必验项全部通过后，模块才能标记为“通过”；可选项不影响 P0 交付。测试中发现的实际结果如果与文档不一致，应记录为缺陷，不能临时修改预期结果来获得通过。

## 2. 验收范围和模块关系

| 模块 | 名称 | 主要产物 | 前置模块 |
| --- | --- | --- | --- |
| M01 | 项目骨架与主界面 | 可构建、可启动的 WPF 应用 | 无 |
| M02 | 白名单目录表与安全边界 | 普通/演示白名单、路径安全检查 | M01 |
| M03 | 缓存扫描 | 异步扫描、统计、进度、取消和容错 | M02 |
| M04 | 结果展示与项目选择 | 分类结果、默认选择、预计空间 | M03 |
| M05 | 确认与缓存清理 | 二次确认、仅清理所选文件、异常隔离 | M02、M04 |
| M06 | 清理报告与重新扫描 | 报告、部分完成、磁盘刷新、闭环流程 | M05 |
| M07 | 生命周期与稳定性 | 关闭保护、重复操作和异常状态 | M03、M05、M06 |

M02 和 M05 是安全关键模块。任何路径越界、未确认删除、真实目录被自动化测试修改或程序静默提权的现象，均直接判定当前版本不可交付。

### 3.2 通用准备

在仓库根目录执行：

```powershell
dotnet restore SystemCacheCleaner.sln
dotnet build SystemCacheCleaner.sln -c Debug --no-restore
dotnet test SystemCacheCleaner.sln -c Debug --no-build
```

手工清理验收必须使用演示模式。先运行：

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\Prepare-DemoData.ps1
dotnet run --project .\src\SystemCacheCleaner\SystemCacheCleaner.csproj -- --demo
```

普通模式只用于验证目录显示、扫描、容错和风险提示，默认不得点击最终确认执行真实清理。确需验证普通模式删除时，必须由项目负责人单独授权并提前备份；该操作不属于本文档的默认验收步骤。

### 3.3 证据格式

自动化用例应保留控制台输出，至少包含执行命令、总用例数、通过数和失败数。手工用例按需保存截图，截图文件建议采用 `artifacts/acceptance/<日期>/<用例ID>.png`；涉及文件系统变化的用例，还应记录操作前后的文件清单或数量。验收记录应使用以下格式：

```text
用例 ID：AC-Mxx-xx
结果：通过 / 未通过 / 受阻
实际表现：
证据位置：
缺陷编号或备注：
```

“看起来正常”不能作为证据。统计类用例要记录期望数值和实际数值，删除类用例要证明已选文件删除且未选文件仍然存在。

### 3.4 逐模块提交与独立审查

开发 Agent 完成一个模块后，只能把模块状态更新为“待验收”，不能自行标记“通过”。提交审查时应提供模块 ID、修改文件清单、实现摘要、构建与测试命令输出、已执行的手工用例和已知问题。缺少这些信息时，审查仍会检查代码，但证据缺口会记录为“未验证”。

审查按固定顺序进行：先核对修改范围和安全边界，再检查实现与文档是否一致，然后执行构建、自动化测试和该模块的必验用例，最后执行受影响模块的回归测试。审查结论采用以下格式：

```text
模块：Mxx
结论：通过 / 需修改 / 受阻
已通过用例：
未通过或未验证用例：
发现的问题：按 P0、P1、P2、P3 分级
必须修改项：
建议修改项：
复验范围：
```

结论为“需修改”时，开发 Agent 应只修复列出的缺陷及其直接影响，不借复验机会扩展功能。修复后重新提交，审查至少复验失败用例、修改代码对应的自动化测试及受影响的安全回归项。只有独立审查结论为“通过”，才能更新模块汇总表并开始依赖该模块的后续工作。
