# 系统缓存清理工具软件 V1.0 实现计划

## 1. 文档用途

本文档用于指导开发 Agent 将《系统缓存清理工具软件 V1.0 需求文档》实现为可运行、可演示、可验收的 Windows 桌面程序。执行者应同时阅读需求文档和《模块验收文档》，不得只根据界面效果推断功能是否完成。

本轮开发固定采用 **C#、.NET 8 和 WPF**。V1.0 只实现需求中的 P0 功能，并补充一个用于安全演示和自动化验证的演示模式。操作历史、主题切换、计划任务、注册表清理、系统优化等 P1 或范围外能力不在本轮实现范围内。

## 2. 交付目标与完成定义

项目完成时，应同时具备以下成果：

- Windows 10/11 上可启动的 WPF 单窗口程序，界面完整显示“系统缓存清理工具软件 V1.0”。
- 一套完整的“扫描—选择—确认—清理—报告—重新扫描”流程。
- 普通模式下只扫描三个预设白名单类别；演示模式下只处理固定的演示目录。
- 自动化测试覆盖扫描统计、选择清理、越界保护、异常跳过、取消和重复扫描。
- 可执行的演示数据准备脚本、构建说明和使用说明。
- 构建、测试和模块验收均通过，验收证据符合《模块验收文档》的要求。

“代码已经编写”不等于完成。只有对应模块的必验项通过，且没有 P0 阻断缺陷，才能将该模块标记为完成。

## 3. 固定范围与实现假设

### 3.1 目标环境

| 项目 | 固定选择 |
| --- | --- |
| 操作系统 | Windows 10 或 Windows 11，x64 |
| SDK | .NET 8 SDK |
| 桌面框架 | WPF |
| 目标框架 | 应用项目和测试项目均为 `net8.0-windows` |
| 运行时依赖 | 不引入第三方运行时 NuGet 包 |
| 测试框架 | MSTest；测试项目可使用 Microsoft.NET.Test.Sdk 等测试依赖 |
| 权限 | 默认以普通用户运行，不自动请求管理员权限 |
| 界面语言 | 简体中文 |
| 窗口形态 | 单主窗口，确认操作可使用模态对话框 |

如果执行环境没有 .NET 8 SDK，应记录环境阻塞，不得擅自更换为其他语言或框架。若模板生成的项目文件与本计划略有差异，以能在 .NET 8 下稳定构建的最小配置为准。

### 3.2 普通模式的扫描白名单

普通模式只允许扫描下列类别，不提供任意路径输入框：

| 类别 ID | 显示名称 | 路径规则 | 默认选择 | 风险说明 |
| --- | --- | --- | --- | --- |
| `user-temp` | 用户临时文件 | `Path.GetTempPath()` | 是 | 可能包含正在使用的文件，清理时自动跳过 |
| `windows-temp` | Windows 临时文件 | `%WINDIR%\Temp` | 否 | 部分文件需要管理员权限，默认不选中 |
| `edge-cache` | Microsoft Edge 缓存 | `%LOCALAPPDATA%\Microsoft\Edge\User Data\Default\Cache\Cache_Data` | 是 | 浏览器运行时部分文件可能被占用 |

目录不存在时，该类别仍应出现在结果中，显示 0 个文件和“目录不存在，已跳过”，不能把它当成程序错误。V1.0 不枚举 Edge 的其他 Profile，不扫描 `Downloads`、用户文档、回收站、注册表或 Windows Update 目录。

### 3.3 演示模式

程序接受固定命令行参数 `--demo`。演示模式把三个类别映射到：

```text
%LOCALAPPDATA%\SystemCacheCleaner\DemoCache\
├── UserTemp\
├── WindowsTemp\
└── EdgeCache\
```

演示模式必须在主界面持续显示“演示模式：当前数据不是真实系统缓存”。`tools/Prepare-DemoData.ps1` 用于在上述三个目录中创建可识别的测试文件。脚本不得读取、覆盖或删除真实缓存，也不得接受任意清理路径参数。演示模式和普通模式共用扫描与清理服务，只替换白名单目录定义，避免形成两套业务逻辑。

## 4. 安全边界

清理功能必须满足以下约束，任何一项缺失都属于 P0 阻断问题：

1. 扫描阶段只读取目录和文件元数据，不修改文件、目录属性或内容。
2. 清理输入只能来自当前一次扫描形成的结果快照，不能在清理时重新扩展扫描范围。
3. 每个文件删除前都要重新取得绝对路径，并使用不区分大小写、包含目录分隔符边界的方式确认其位于所属白名单根目录下。
4. 白名单根目录本身不得被删除；V1.0 只删除文件，不递归删除目录。
5. 扫描时跳过符号链接、junction 和其他带 `ReparsePoint` 属性的目录或文件，不跟随重解析点进入其他位置。清理前还应检查从白名单根目录到文件父目录之间的已有路径段，拒绝经过重解析点的路径。
6. 文件消失、被占用、无权限、路径过长或单个目录不可访问时，记录为跳过或失败并继续处理其余项目。
7. 普通模式不得创建待清理文件；演示数据只能由明确执行的准备脚本创建。
8. 程序不得静默提权，不得调用 Shell 或外部清理命令，不得把用户输入拼接为删除路径。

### 5.1 模型定义

模型应保持简单并能表达扫描快照和清理结果：

- `CacheCategoryDefinition`：类别 ID、名称、说明、根目录集合、是否默认选择、风险级别。
- `CacheFileItem`：绝对路径、所属类别 ID、扫描时文件长度、扫描状态。扫描结果建立后不允许 ViewModel 任意改写路径。
- `CategoryScanResult`：类别定义、文件快照、文件数、总字节数、是否选中、状态说明和警告摘要。
- `ScanSessionResult`：开始/结束时间、是否取消、类别结果集合和全局异常摘要。
- `CleanupFileResult`：文件路径、扫描时长度、成功/跳过/失败状态、用户可理解的原因。
- `CleanupSessionResult`：开始/结束时间、成功数、跳过数、失败数、成功删除文件的字节总和和总体状态。

“实际释放空间”在 V1.0 中定义为成功删除文件在扫描快照中的长度之和。磁盘可用空间仍需在清理后重新读取并刷新，但不以磁盘前后差值作为报告值，因为系统中的其他进程可能同时写入或删除文件。

### 5.2 服务接口

`ICacheCatalog` 只负责返回普通模式或演示模式的固定类别定义。`ICacheScanService` 接收类别定义、`CancellationToken` 和进度回调，返回完整或已取消的扫描快照。`ICleanupService` 接收当前扫描快照中被选中的类别结果，逐文件执行安全检查和删除。`IDiskSpaceService` 读取系统盘总容量和可用容量。

服务不得直接更新 WPF 控件。进度通过 `IProgress<T>` 或等价的轻量机制回传，界面状态和命令可用性由 `MainViewModel` 统一管理。项目不引入依赖注入框架，依赖可在 `App.xaml.cs` 或窗口初始化处显式组装。

## 6. 界面与状态设计

主窗口建议尺寸为 1180×720，最小尺寸为 960×600。使用 `Segoe UI`、浅灰背景、白色内容区域和一种蓝色主色，信息层级主要依靠字号、间距和字重表达，不堆叠无意义卡片或动画。

窗口从上到下分为：标题与版本区、系统盘信息和当前状态区、主要内容区、底部操作区。扫描完成后，主要内容区使用表格或列表显示选择框、类别、状态、文件数量、占用空间和风险说明。底部持续显示“已选择 N 类，预计可释放 X”，并提供与状态匹配的主要按钮。

界面状态固定为：

```text
Idle
  └─开始扫描→ Scanning
Scanning
  ├─扫描完成→ ScanCompleted
  └─用户取消→ ScanCompleted（标记“结果不完整”）
ScanCompleted
  ├─重新扫描→ Scanning
  └─立即清理→ Confirmation
Confirmation
  ├─取消→ ScanCompleted
  └─确认→ Cleaning
Cleaning
  └─任务结束→ CleanupCompleted / PartialCompleted
CleanupCompleted / PartialCompleted
  └─重新扫描→ Scanning
```

状态切换时必须同步更新命令：扫描期间禁止再次扫描和清理；没有选择或预计空间为 0 时禁用“立即清理”；清理期间禁止改变选择和重复清理。确认对话框要显示所选类别和预计空间，并明确说明被占用或无权限文件会被跳过。

扫描进度在总文件数未知时按“已完成类别数/总类别数”展示，同时显示当前类别和已发现文件数，不伪造精确百分比。清理进度可按“已处理文件数/待处理文件数”计算。

关闭窗口时，空闲状态直接退出；扫描中关闭应提示并取消扫描；清理中关闭应提示“停止后已删除的文件无法恢复”。用户确认后请求取消，等待当前文件处理完成再退出，不允许在删除调用执行到一半时强制终止进程。

## 7. 分模块实施任务

每个模块只处理本模块范围，完成后立即执行相应构建、测试和手工验收。后续模块不得以“最终一起修复”为由跳过当前检查。以下编号与《模块验收文档》完全一致。

### M01：项目骨架与可启动界面

创建解决方案、WPF 项目和 MSTest 项目，设置项目引用，完成主窗口静态结构和基础 MVVM 工具。程序启动后显示软件全名、版本、系统盘信息占位、空状态说明和“开始扫描”按钮。

参考命令：

```powershell
dotnet new sln -n SystemCacheCleaner
dotnet new wpf -n SystemCacheCleaner -o src/SystemCacheCleaner
dotnet new mstest -n SystemCacheCleaner.Tests -o tests/SystemCacheCleaner.Tests
dotnet sln SystemCacheCleaner.sln add src/SystemCacheCleaner/SystemCacheCleaner.csproj
dotnet sln SystemCacheCleaner.sln add tests/SystemCacheCleaner.Tests/SystemCacheCleaner.Tests.csproj
dotnet add tests/SystemCacheCleaner.Tests/SystemCacheCleaner.Tests.csproj reference src/SystemCacheCleaner/SystemCacheCleaner.csproj
```

模板生成后应立即确认两个项目的 `TargetFramework` 均为 `net8.0-windows`，应用项目同时保留 `<UseWPF>true</UseWPF>`。测试项目若仍为 `net8.0`，应先改正再添加项目引用。

完成标准：`dotnet build SystemCacheCleaner.sln` 成功，程序可启动和正常关闭；窗口在 100%、125% 和 150% 缩放下无文字遮挡或按钮越界。

### M02：白名单目录表与安全边界

实现普通/演示白名单目录表和 `PathSafety`。路径安全检查必须覆盖绝对路径规范化、目录分隔符边界、根目录自身、相似前缀路径、`..` 规范化越界以及从白名单根目录到文件父目录之间的重解析点。普通模式和演示模式只能替换目录定义，不能修改扫描与清理规则。

完成标准：普通模式恰好提供约定的三个类别；演示模式只映射到固定 DemoCache 子目录；白名单内路径、相似前缀越界、根目录、`..` 和重解析点测试符合安全预期。

### M03：缓存扫描

实现按目录逐层容错枚举、扫描快照、进度反馈和取消逻辑。每次新扫描先清空旧结果，不能累计旧数据。目录不存在、单个文件读取失败和单个子目录无权限只影响对应项目。

扫描不得读取文件内容。需要统计的只有路径、长度、所属类别和状态。遍历目录前检查重解析点，避免越过白名单边界。

完成标准：空目录、多文件、嵌套目录、目录不存在、取消扫描和连续两次扫描的自动化测试通过；演示模式可看到三个类别及明确标识。

### M04：扫描结果选择与预计空间

将类别结果绑定到界面，支持逐类别勾选。默认选择来自 `CacheCategoryDefinition`，不依据类别名称写死在 XAML 中。预计空间等于当前选中类别的扫描字节数之和；选择变化后立即刷新。

扫描被取消时保留已完成和已统计的结果，并显示“扫描已取消，结果可能不完整”。用户仍可查看结果，但为了避免基于不完整快照删除文件，“立即清理”保持禁用，只有重新完整扫描后才能清理。

完成标准：默认选中状态正确；全选、取消选择、0 字节和扫描取消时的按钮状态及汇总数值正确。

### M05：确认、安全清理与异常隔离

实现清理确认对话框和 `CleanupService`。确认前不允许删除文件。清理只消费当前扫描快照中已选类别的文件列表，每个文件删除前执行白名单归属检查和重解析点检查。

文件删除成功后记录成功和扫描长度；文件已不存在记录为跳过；路径越界、重解析点或安全检查不通过记录为失败；被占用、无权限及其他可恢复 I/O 异常记录具体原因并继续。取消信号到达后不再开始新的文件删除，已完成结果保留并生成“部分完成”报告。

完成标准：未选文件不变、已选文件被删除、锁定文件被跳过、构造的越界路径被拒绝，整个任务不会因单个文件异常退出。

### M06：清理报告、磁盘刷新和重复流程

报告显示开始/结束时间、成功数、跳过数、失败数、实际清理文件大小和失败原因摘要。全部目标处理成功显示“清理完成”；存在跳过、失败或用户取消时显示“部分完成”，不得使用全成功文案。

清理结束后重新读取系统盘信息。界面提供“重新扫描”，重新扫描得到的数据应与当前文件状态一致。报告中可保留本次结果，但 V1.0 不要求跨进程保存操作历史。

完成标准：报告统计与文件系统结果一致；重新扫描不包含已经删除的文件；部分失败状态和摘要正确。

### M07：生命周期与稳定性

实现窗口关闭保护和连续任务状态恢复。扫描中关闭时提示用户并通过取消令牌停止；清理中关闭时说明已删除文件无法恢复，用户确认后停止启动新的删除操作，等待当前操作结束再退出。所有异步命令都应捕获并转换可恢复异常，任务结束后恢复正确的按钮状态。

连续扫描、取消后重扫、清理后再次准备演示数据并重扫时，必须创建新快照和新取消令牌，不能复用已取消令牌或陈旧结果。窗口退出后不应保留后台扫描或清理任务。

完成标准：关闭扫描、关闭清理、连续三次扫描、连续两轮清理和服务异常注入验收通过；程序退出后没有残留进程或继续修改文件的后台任务。

### M08：演示脚本、说明文档与发布构建

实现 `Prepare-DemoData.ps1`，在固定演示根目录的三个子目录中创建若干不同大小的 `.tmp`、`.cache` 或 `.dat` 文件。脚本应可重复执行，文件命名稳定，并在执行结束时打印目录、文件数和总大小。不得自动启动清理，也不得触碰演示根目录以外的位置。

补充 `README.md` 和操作手册，记录普通模式、演示模式、构建、测试和发布步骤。发布命令建议为：

```powershell
dotnet publish src/SystemCacheCleaner/SystemCacheCleaner.csproj -c Release -r win-x64 --self-contained false -p:PublishSingleFile=true
```

若实际发布参数发生变化，应同步更新 README 和操作手册。完成标准：从干净构建开始可按文档启动程序；演示脚本、扫描、选择、确认、清理、报告和重新扫描流程能连续完成。

## 8. 自动化测试最低集合

测试必须使用测试框架提供的临时目录或 `%TEMP%` 下新建的唯一测试目录。每个测试只删除自己创建且已确认位于该临时目录下的数据。测试完成后清理临时数据；即使清理失败，也不得扩大删除范围。

| 测试 ID | 测试目标 | 关键断言 |
| --- | --- | --- |
| UT-SCAN-01 | 空目录扫描 | 文件数为 0、字节数为 0、无未处理异常 |
| UT-SCAN-02 | 多层目录扫描 | 文件数和总长度与准备数据一致 |
| UT-SCAN-03 | 目录不存在 | 类别保留、结果为 0、状态说明正确 |
| UT-SCAN-04 | 取消扫描 | 返回取消状态，不再继续枚举，进程不异常 |
| UT-SCAN-05 | 重复扫描 | 第二次结果反映当前文件状态，不与第一次累计 |
| UT-SAFE-01 | 白名单内路径 | 安全检查通过 |
| UT-SAFE-02 | 相似前缀越界路径 | 如 `Root2` 不得被视为位于 `Root` 内 |
| UT-SAFE-03 | 根目录自身 | 不得作为可删除文件 |
| UT-CLEAN-01 | 只清理所选类别 | 已选文件删除，未选文件保留 |
| UT-CLEAN-02 | 文件已不存在 | 记录跳过，后续文件继续处理 |
| UT-CLEAN-03 | 文件被独占锁定 | 记录失败或跳过，服务不抛出到界面层 |
| UT-CLEAN-04 | 注入越界文件 | 拒绝删除并记录安全原因 |
| UT-CLEAN-05 | 清理取消 | 收到取消后不开始新的删除，生成部分报告 |
| UT-VM-01 | 无选择时命令状态 | “立即清理”不可执行 |
| UT-VM-02 | 选择变化 | 预计空间和命令状态即时更新 |
| UT-VM-03 | 扫描/清理运行中 | 重复命令不可执行，结束后恢复正确状态 |

对于“无权限目录”这类依赖系统 ACL 且在不同机器上不稳定的情形，自动化测试可通过可控异常注入验证容错，真实权限表现通过手工验收补充。不得为了制造无权限场景而修改真实系统目录权限。

## 9. Agent 执行与交接规则

接手开发的 Agent 应遵守以下工作方式：

1. 开工前阅读需求文档、实现计划和模块验收文档，说明本轮要实现的模块 ID。
2. 只修改当前模块直接需要的文件，不顺带实现 P1 功能或重构无关代码。
3. 先建立或补充对应测试，再完成实现；至少保证本模块关键安全路径有自动化验证。
4. 每个模块结束时执行 `dotnet build SystemCacheCleaner.sln` 和 `dotnet test SystemCacheCleaner.sln`，再执行该模块的手工验收项。
5. 交接记录必须包含：模块 ID、修改文件、实现内容、执行命令、通过/失败结果、截图或日志位置、尚未解决的问题。开发 Agent 提交时只能把模块标记为“待验收”，由独立审查方依据验收文档决定是否通过。
6. 如果实现与本计划冲突，不得静默改变需求。应记录冲突、影响和最小修订建议，交由项目负责人确认。
7. 禁止在自动化测试或无人监督的验收中对普通模式真实缓存执行清理。涉及删除的演示一律使用 `--demo` 模式或测试临时目录。

推荐按 M01 至 M08 的顺序交接模块；模块编号和验收门槛见《模块验收文档》。M02 至 M05 涉及核心数据和安全边界，后续 Agent 修改这些模块时必须重新执行相关回归用例。

## 10. 最终质量门槛

发布候选版本需同时满足：Debug 和 Release 构建成功；全部自动化测试通过；M01—M08 的必验项通过；没有未经解释的程序崩溃、路径越界删除或错误的“全部成功”提示；README、操作手册、界面文案与实际实现一致。

若某项受环境限制无法执行，应在验收记录中标记“未验证”，写明环境、原因和复现命令。未验证不能写成通过，涉及安全清理、确认流程或路径越界保护的项目不得带着“未验证”状态交付。
